# pyjwt Commit 9f36c24 README Update

## Summary
This commit standardizes JWK output formatting across all algorithm classes in `pyjwt.algorithms` by enforcing dictionary returns and removing string serialization capabilities. The changes affect base class definitions and implementations for HMAC, RSA, EC, OKP, and None algorithms, breaking backward compatibility for any code relying on string-based JWK outputs.

---

## Changed Modules
- `jwt/algorithms.py` (core module containing all algorithm classes)

---

## Affected Functions

### `jwt.algorithms.Algorithm.to_jwk`
**Change**: Enforced dictionary-only output by requiring `as_dict=True` parameter  
**Why**: To standardize JWK serialization format and simplify type handling  
**Impact**:  
- Callers must explicitly pass `as_dict=True`  
- Code relying on default behavior (`as_dict=False`) returning strings will break  
- Base class change forces all subclasses to adopt dictionary output

---

### `jwt.algorithms.HMACAlgorithm.to_jwk`
**Change**: Removed string serialization path, now dictionary-only return  
**Why**: Align with base class requirements and ensure consistent output  
**Impact**:  
- Existing code expecting JSON strings when `as_dict=False` will fail  
- Consumers must update to handle dictionary outputs and explicitly set `as_dict=True`

---

### `jwt.algorithms.RSAAlgorithm.to_jwk`
**Change**: Enforced dictionary output requirement through updated type hints  
**Why**: Maintain consistency with base class contract  
**Impact**:  
- Breaks backward compatibility for string-based JWK consumers  
- Requires migration to dictionary handling in calling code

---

### `jwt.algorithms.ECAlgorithm.to_jwk`

**Why**: Ensure uniform behavior across all algorithm implementations

---

## Overall Impact Analysis
This commit represents a significant API breaking change that:
1. **Standardizes Output**: Eliminates format ambiguity by enforcing dictionary returns across all JWK serialization operations
2. **Improves Type Safety**: Removes conditional logic for string vs dict returns, simplifying type handling
3. **Breaks Backward Compatibility**: Requires all consumers to:
   - Explicitly pass `as_dict=True`
   - Update code to handle dictionary outputs instead of strings
4. **Future-Proofs API**: Establishes consistent behavior that simplifies maintenance and reduces edge cases

Consumers should audit all JWK-related code paths and update implementations to handle dictionary outputs. The changes work together to create a more robust and predictable JWK serialization interface at the cost of requiring migration effort for existing integrations.
